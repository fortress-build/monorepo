A Capability Statement documents a set of capabilities (behaviors) of a FHIR Server or Client for a particular version of FHIR that may be used as a statement of actual server functionality or a statement of required or desired server implementation.

## Scope and Usage

The CapabilityStatement is a key part of the FHIR conformance framework. It serves as:
- A statement of actual software features
- A set of rules for application functionality
- A connection to detailed functionality statements (StructureDefinitions and ValueSets)

It can be used for:
- System compatibility testing
- Code generation
- Conformance assessment

### Types of Capability Statements

There are three main types of Capability Statements:

#### 1. Instance
- **Purpose**: Describes an actual implementation
- **Requirements**: Implementation must be present, software may be present
- **Use Cases**:
  - Describes capabilities of deployed solutions at specific access points
  - Used by FHIR RESTful solutions for the capabilities operation
  - Forms basis for testing and certification of specific installations

#### 2. Capability
- **Purpose**: Describes software solution capabilities
- **Requirements**: Implementation must be absent, software must be present
- **Use Cases**:
  - Documents generic capabilities of software applications
  - Used as marketing tool to describe vendor capabilities
  - Basis for software conformance testing
  - May describe multiple possible configurations

#### 3. Requirements
- **Purpose**: Describes desired system capabilities
- **Requirements**: Both implementation and software must be absent
- **Use Cases**:
  - Used in architectural design to document needed capabilities
  - Part of RFP process to document requirements
  - Can include additional documentation/extensions to specify expectations

### Working with Capability Statements

#### Combined Usage
These three types can work together:
- Requirements statements can be compared against solution statements in RFPs
- Solution statements for software packages inform implementation statements for specific installations

#### Practical Considerations
- Capability Statements enable some automatic configuration and adaptation
- Complete capture of all system variations is rarely practical
- They serve as both:
  - An automation tool
  - A source of human-readable documentation

#### Version Support
- Applications may implement multiple FHIR versions
- Each version requires its own CapabilityStatement
- Systems can use the `$versions` operation for version management

#### Normative Content
- Core CapabilityStatement resource is Normative
- Many operational flags are marked as trial-use
- Applications can request 'Normative content only' using the mode parameter
- Future versions may introduce changes to trial-use portions based on community feedback

## Background and context
Capability Statements provide for a degree of automatic configuration and adaptation. However, capturing absolutely every variation that could impact the interoperability of two systems, let alone keeping that detailed information up-to-date as systems evolve through maintenance and upgrades, is rarely practical. Therefore, capability statements should be seen as an interim step. They provide a degree of automation. However, they also provide a great deal of human-readable content that can minimize the need for direct communication between the operators of the systems being configured to interoperate.

5.3.2.1 Supporting Multiple Versions 
Applications may implement multiple versions. If they do, then a CapabilityStatement describes the system's support for a particular version of FHIR, and the system will have multiple statements, one for each version it supports. For further information, see Managing Multiple Versions, and the $versions operation.

5.3.2.2 Mixed Normative Content 
While the core of the CapabilityStatement resource is Normative, many of the flags that indicate exactly how the system operates are marked as trial-use. Roughly, the portions of the resource that correspond to OpenAPI document icon elements are normative.

Applications looking for normative stability should only use the normative parts of the resource, and not populate or ignore the portions labelled trial-use. To assist with this, clients can ask for the server to return a 'Normative content only' CapabilityStatement using the mode parameter on /metadata.

Community discussion regarding more capable, efficient and computable representations of an applications capabilities may lead to change to the trial-use parts of this resource or the creation of new resources and/or functionality in future versions of this specification.
